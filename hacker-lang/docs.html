HackerOS — język skryptowy

Hacker Lang

Interpretowany język programowania dla HackerOS (Debian-based).
Własna składnia operatorowa, wbudowany linter, REPL, kompilator bytecode+Cranelift, manager pakietów bit.

wbudowany w HackerOS tylko dla HackerOS pliki .hl / .bc gen 1
01

Czym jest Hacker Lang?

Hacker Lang (HL) to interpretowany język programowania napisany w Rust, zaprojektowany jako natywny język skryptowy systemu HackerOS. Pliki źródłowe mają rozszerzenie .hl, skompilowany bytecode .bc.

HL łączy własną składnię operatorową z bezpośrednim dostępem do powłoki systemowej, wbudowanym linterem, bibliotekami standardowymi (main/net, main/sys…), ponad 40 quick-funkcjami, prawdziwym kompilatorem, managerem pakietów bit oraz goroutines/channels inspirowanymi Go.

✨
Nowe w gen 1: kompilacja dwuetapowa (.hl → .bc → ELF), operator & (tło), *> (hsh), _N (pętla N razy), << plik.hl (import pliku), goroutines (:*), channels (:**), manager pakietów bit, nowe przestrzenie nazw importów (main/, bit/, github/), nowe biblioteki (main/colors, main/cli, main/progress-bar, main/json, main/hk-parser, main/hacker).

Szybki start

hello.hl
#!/usr/bin/env hl
/// Mój pierwszy skrypt w Hacker Lang
using <gen 1>

// curl
# <main/net>
# <main/colors>

% name    = HackerOS
% version = gen 1

~> Witaj w @name @version!

: greet def
    ~> Kernel:
    > uname -r
    ::upper @name
done

-- greet

;; uruchom serwer w tle
& python3 -m http.server 8080

? ok
    ::green System gotowy ✓
done
URUCHAMIANIE I KOMPILACJA
hl hello.hl              # uruchom skrypt
hl compile hello.hl      # .hl → hello.bc (bytecode)
hl compile hello.bc      # .bc → binarka ELF
./hello.bc               # uruchom .bc bezposrednio (shebang)
hl run hello.bc          # uruchom .bc przez hl
hl repl                  # interaktywna powloka REPL
hl check hello.hl        # sprawdz skladnie + linter

Hsh Shell — *>

Operator *> uruchamia komendę przez hsh -c "..." — autorską powłokę HackerOS.

SKŁADNIA *>
*> ls -la
*> cat /etc/os-release
*> echo "Hello from hsh"  ;; echo dozwolone w hsh

;; *> nie obsluguje interpolacji @zmiennych — uzyj >> dla tego celu
💡
*> przekazuje komendę verbatim do hsh -c. Nie obsługuje interpolacji @zmiennych — użyj >> gdy potrzebujesz zmiennych HL. echo jest dozwolone wewnątrz *> (bo to kontekst hsh, nie HL).
10

Tło — &

Operator & uruchamia komendę w tle — nie czeka na jej zakończenie. PID jest dostępny przez @_bg_pid.

SKŁADNIA &
& python3 -m http.server 8080
& redis-server --port 6380

;; Równoległe pobieranie
& wget -q https://example.com/plik1.zip
& wget -q https://example.com/plik2.zip
& wget -q https://example.com/plik3.zip

;; PID ostatniego procesu w tle
~> PID: @_bg_pid

;; Działa też ze zmiennymi HL (interpolacja)
% port = 9000
& python3 -m http.server @port

Goroutines i Channels — :* :**

Goroutines (:*) i channels (:**) to mechanizm współbieżności inspirowany Go. Goroutine uruchamia blok kodu w osobnym wątku. Channel służy do komunikacji między goroutines.

💡
W trybie kompilacji (.bc → ELF) goroutines i channels są pomijane — działają tylko w trybie interpretowanym. Dla pełnej współbieżności w skompilowanych binkarkach użyj operatora &.
SKŁADNIA GOROUTINE / CHANNEL
;; Goroutine — uruchom blok w tle (wg. Go goroutines)
:*
    > python3 -m http.server 9090
done

;; Channel — komunikacja miedzy goroutines
:** wyniki

:*
    > nmap -sn 192.168.1.0/24
    *-- wyniki
done

:*
    > masscan -p1-1000 192.168.1.0/24
    *-- wyniki
done

;; Odbierz z kanalu
*-- wyniki
SymbolNazwaOpis
:*GoroutineUruchom blok w osobnym wątku
:**ChannelZadeklaruj kanał komunikacji
*--Channel OpWyślij/odbierz przez kanał

Manager pakietów — bit

bit to manager pakietów napisany w Hacker Lang dla ekosystemu HL. Lista paczek: https://github.com/bit-io/repository/blob/main/bit-repo/repo-list.json

📦
Wywołanie bit bez argumentów wyświetla pomoc.
KOMENDY bit
bit                           # wyswietl pomoc
bit install <nazwa>           # zainstaluj pakiet
bit remove  <nazwa>           # usun pakiet
bit list                      # pokaz dostepne pakiety
bit update                    # zaktualizuj liste pakietow
bit info    <nazwa>           # informacje o pakiecie
STRUKTURA PROJEKTU BIT
;; Struktura katalogu z plikami bit:
projekt/
  build.hl          # glowny skrypt budowania
  main.hl           # kod zrodlowy
  .cache/           # izolowane srodowisko (tworzone przez bit)
    repo-list.json  # lista paczek
    hashlib/        # pobrane biblioteki
  main               # binarka wynikowa (kolo build.hl)

;; Lub Rust-like (source-code/):
projekt/
  source-code/      # kod zrodlowy
  build.hl          # dodatkowe informacje
  .cache/           # izolowane srodowisko (usuwane po zakonczeniu)

;; Lub run-only (bez kompilacji):
projekt/
  run.hl            # skrypt uruchamiany (nie kompilowany)
  .cache/           # izolowane srodowisko

Kompilacja — .hl → .bc → ELF

Kompilacja jest dwuetapowa:

.hl → .bc

Etap 1: parsuj .hl, serializuj AST do JSON bytecode .bc. Plik .bc jest wykonywalny (ma shebang).

.bc → ELF

Etap 2: wczytaj .bc, lower AST → HlIR → Cranelift → obiekt + C runtime → binarka ELF / .so.

PRZEPŁYW KOMPILACJI
hl compile skrypt.hl         # → skrypt.bc  (bytecode)
./skrypt.bc                  # → uruchom bezposrednio (shebang)
hl run skrypt.bc             # → uruchom przez hl
hl compile skrypt.bc         # → skrypt      (binarka ELF)
hl compile --shared skrypt.bc # → skrypt.so   (biblioteka bit)
./skrypt                     # → uruchom binarke

// nmap
// curl

# <main/colors>

% target = 192.168.1.1
% ports  = 1-1024

::hr 50
::bold Network Audit gen 1

: port_scan def
    ~> Skanuje porty @ports na @target...

: probe_isolated def

done

-- probe_isolated

;; Rownolegly skan w tle
