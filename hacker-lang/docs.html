Hacker Lang
Dokumentacja

Skryptowy język powłoki zaprojektowany pod HackerOS. Łączy czytelność shellscriptów z własną, minimalistyczną składnią. Przeznaczony do automatyzacji, narzędzi CLI i systemowych skryptów. Wspiera zaawansowane funkcje jak klasy, enumy i pluginy dla maksymalnej elastyczności.

✓ Pluginy i biblioteki ✓ Klasy i funkcje ✓ Bezpieczeństwo i izolacja
02

Struktura pliku .hl

Każdy plik hacker-lang ma rozszerzenie .hl. Interpreter przetwarza linie jedna po drugiej. Kolejność sekcji jest elastyczna, ale konwencjonalnie wygląda tak:

hacker-lang · struktura.hl 
! Zależności systemowe (opcjonalne)
// curl
// jq
! Biblioteki (opcjonalne)
#<source/utils>
#<core/io:1.0>
! Zmienne globalne
@APP_NAME="MójSkrypt"
@VERSION="1.0.0"
! Definicje funkcji
:powitaj def
  log "Witaj w hacker-lang!"
done
! Ciało główne (main)
.powitaj
end 0

Konwencje struktury

Chociaż kolejność jest elastyczna, zaleca się umieszczanie zależności na górze, następnie zmiennych, definicji i ciała głównego na końcu. To ułatwia analizę statyczną PLSA.

ℹ
Ważne: Puste linie są ignorowane. Każda niepusta linia to jedna instrukcja — hacker-lang nie używa średników ani nawiasów klamrowych jako separatorów bloków. Użyj komentarzy do organizacji kodu.
03

Komentarze

Hacker-lang obsługuje dwa rodzaje komentarzy: liniowe i blokowe. Komentarze są ignorowane przez interpreter, ale analizowane przez PLSA do dokumentacji.

SkładniaTypOpis
! tekst liniowy Komentarz jednolinijkowy — cała linia jest ignorowana
!! blokowy Przełącza tryb komentarza blokowego (otwiera/zamyka)
komentarze.hl 
! To jest komentarz liniowy
@AUTOR="Jan" ! komentarz w tej samej linii — NIEOBSŁUGIWANY (! musi być na początku)
!!
To jest blok komentarza.
Może zajmować wiele linii.
Żadna linia tutaj nie jest parsowana.
!!
log "Ten kod jest wykonywalny"

Przykłady użycia komentarzy

Komentarze blokowe są przydatne do wyłączania dużych fragmentów kodu podczas debugowania.

debug.hl 
!!
Ten kod jest tymczasowo wyłączony
> rm -rf /tmp/test
!!
⚠
Uwaga: Komentarze liniowe (!) muszą być na początku linii (po trim). Nie można dodać komentarza na końcu linii z kodem. Użyj blokowych dla wieloliniowych wyjaśnień.
04

Zmienne

Hacker-lang rozróżnia dwa rodzaje zmiennych: globalne (env) oznaczone prefiksem @ i lokalne bez prefiksu. Zmienne "surowe" (raw) oznaczamy prefiksem ~. Zmienne mogą być blokowane dla bezpieczeństwa.

@ — Zmienne globalne (env)
Ustawiane jako zmienne środowiskowe. Dostępne przez cały program. Konwencja: WIELKIE_LITERY. Mogą być dziedziczone przez subshelle.
name= — Zmienne lokalne
Zmienne lokalne funkcji lub głównego ciała. Nie są eksportowane do środowiska. Przydatne do tymczasowych obliczeń.
~ — Raw (bez podstawienia)
Wartość jest przechowywana bez rozwijania $(...) — surowy tekst. Idealne do szablonów lub komend literalnych.
Odwołania do zmiennych
Zmienne wbudowane shell: $1, $VAR, $(cmd) działają w wartościach lokalnych. Użyj ${VAR} dla złożonych wyrażeń.
zmienne.hl 
! Zmienne globalne (env)
@HOME_DIR="/home/user"
@MAX_RETRY=3
@APP="backup-tool"
! Zmienne lokalne
nazwa_="Jan Kowalski"
wynik_=$(date +%Y%m%d)
liczba_=42
! Zmienna raw — bez interpretacji $(...)
~surowa_="$(to nie zostanie wykonane)"
! Użycie zmiennych w komendach
> echo "Witaj, $nazwa_"
> echo "Data: $wynik_"

Zaawansowane użycie zmiennych

Zmienne mogą być używane w warunkach, pętlach i jako argumenty funkcji. Dla bezpieczeństwa, używaj lock na kluczowych zmiennych.

zmienne-zaawansowane.hl 
@API_URL="https://api.example.com"
lock $API_URL
response_=$(> curl -s $API_URL)
✦
Konwencja nazw: Zmienne lokalne najczęściej kończą się podkreślnikiem (wynik_, plik_) — odróżnia je to od słów kluczowych jak done, end. Używaj raw dla wrażliwych danych.
05

Komendy (polecenia systemowe)

Hacker-lang ma trzy prefiksy do uruchamiania komend systemowych. Różnią się sposobem wykonania i izolacją środowiska. Wszystkie komendy mogą być łączone z ^ dla sudo.

PrefiksTypOpisPrzykład
> komenda RawSub Komenda z podstawianiem zmiennych. Wykonywana w bieżącym kontekście. > echo "Cześć $USER"
|> komenda RawSub Alternatywny prefiks — identyczne działanie co >. |> ls -la /tmp
<| komenda RawSub Alternatywny prefiks (wejście pipe). <| grep "błąd"
>> komenda RawNoSub Komenda bez podstawiania zmiennych. Dosłowny tekst. >> echo '$PATH'
>>> komenda Isolated Komenda w izolowanym środowisku (subshell). >>> mkdir -p /tmp/x
komendy.hl 
@KATALOG="/tmp/moje"
! Zwykła komenda z podstawieniem zmiennych
> mkdir -p $KATALOG
> echo "Tworzę: $KATALOG"
! Bez podstawiania — $KATALOG trafi dosłownie
>> echo 'Dosłownie: $KATALOG'
! Izolowany subshell
>>> cd /tmp && ls -la
! Pipe — alternatywne prefiksy
> cat /etc/passwd
|> grep root

Łączenie komend

Komendy mogą być łączone za pomocą pipe (|) lub operatorów logicznych (&&, ||).

komendy-lancuch.hl 
> ls -la | grep ".log" || echo "Brak logów"
06

Logowanie

Instrukcja log wypisuje wiadomość na standardowe wyjście. Przyjmuje string w cudzysłowie. Logi mogą być kolorowane w terminalu HackerOS.

log.hl 
log "Uruchamiam skrypt..."
log "Wersja: 1.0"
log "Operacja zakończona."

Zaawansowane logowanie

Użyj zmiennych w logach poprzez komendy echo, lub biblioteki logger.

log-dynamic.hl 
status_="sukces"
> echo "Log: operacja $status_"
ℹ
log wymaga cudzysłowów — log "tekst". Bez cudzysłowów nastąpi błąd składni. Użyj > echo tekst dla dynamicznych wartości lub zaawansowanego formatowania.
07

Warunkowe: if / elif / else

Instrukcje warunkowe używają prefiksów ?, ?? i ?:. Komenda do wykonania jest po separatorze >, |> lub <|. Warunki bazują na bash test.

SkładniaOpis
? warunek > komenda Jeśli warunek spełniony — wykonaj komendę
?? warunek > komenda Else-if — jeśli poprzedni warunek nie spełniony
?: > komenda Else — wykonaj jeśli żaden poprzedni warunek
warunki.hl 
wiek_=20
! Podstawowy if
? $wiek_ -ge 18 > echo "Dorosły"
! If / elif / else
? $wiek_ -lt 13 > echo "Dziecko"
?? $wiek_ -lt 18 > echo "Nastolatek"
?? $wiek_ -lt 65 > echo "Dorosły"
?: > echo "Senior"
! Sprawdzenie istnienia pliku
? -f /etc/hosts > echo "Plik istnieje"
?: > echo "Brak pliku"
! Sprawdzenie zmiennej
? -n "$USER" > log "Użytkownik ustawiony"

Złożone warunki

Używaj operatorów logicznych w warunkach: [[ $a -gt 10 && $b = "test" ]].

warunki-zlozone.hl 
? [[ $wiek_ -gt 18 && $status = "aktywny" ]] > echo "Dostęp przyznany"
✦
Warunek to dowolny wyrażenie testowe bash (-f plik, -n "$str", $a -gt $b itp.). Separator > (spacje po obu stronach) oddziela warunek od komendy. Używaj [[ ]] dla złożonych testów.
08

Pętle: loop (N razy)

Pętla =N > komenda powtarza komendę N razy. Prosta i szybka dla stałej liczby iteracji. Może być używana w funkcjach.

loop.hl 
! Powtórz 5 razy
=5 > echo "Iteracja"
! Utwórz 10 plików tymczasowych
=10 > touch /tmp/plik_$(date +%N)
! Ping 3 razy
=3 > ping -c 1 8.8.8.8

Pętla z zmienną

Użyj zmiennej dla dynamicznej liczby iteracji.

loop-dynamic.hl 
@POWTORZEN=4
=$POWTORZEN > echo "Powtórka"
09

Pętla while

Pętla while warunek > komenda powtarza komendę dopóki warunek jest spełniony. Przydatna do czekania na zdarzenia.

while.hl 
licznik_=0
! Pętla — inkrementuj licznik
while $licznik_ -lt 5 > licznik_=$(( $licznik_ + 1 ))
! Czekaj na plik
while ! -f /tmp/gotowy.lock > sleep 1

While z komendą

Użyj while do przetwarzania strumieni.

while-strumien.hl 
while read linia > echo "Linia: $linia" < /etc/passwd
10

Pętla for

Pętla for zmienna in lista > komenda iteruje po elementach listy lub wzorcu glob. Obsługuje argumenty skryptu.

for.hl 
! Iteracja po liście wartości
for kolor in czerwony zielony niebieski > echo "$kolor"
! Iteracja po plikach
for plik in /tmp/*.log > echo "Plik: $plik"
! Iteracja po argumentach skryptu
for arg in $@ > echo "Argument: $arg"

For z zakresem

Użyj seq dla numerycznych zakresów.

for-zakres.hl 
for i in $(seq 1 10) > echo "Numer: $i"
11

Funkcje

Funkcje definiujemy przez :nazwa def i kończymy słowem done. Wywołanie przez .nazwa. Funkcje unsafe (niebezpieczne) oznaczamy ::nazwa def. Funkcje mogą zwracać wartości przez echo.

SkładniaOpis
:nazwa def Deklaracja zwykłej funkcji
::nazwa def Deklaracja funkcji unsafe (operacje systemowe)
done Koniec definicji funkcji
.nazwa Wywołanie funkcji (z kropką na początku)
.klasa.metoda Wywołanie metody klasy
funkcje.hl 
! Definicja zwykłej funkcji
:powitaj def
  imie_="$1"
  > echo "Witaj, $imie_!"
done
! Funkcja unsafe — wykona operacje z podwyższonymi prawami
::wyczysc_logi def
  > rm -f /var/log/app/*.log
  log "Logi wyczyszczone"
done
! Wywołanie funkcji
.powitaj
.wyczysc_logi
! Funkcja z argumentami (dostępne jako $1, $2...)
:suma def
  > echo $(( $1 + $2 ))
done
.suma

Funkcje z zwrotem wartości

Funkcje mogą zwracać wartości przez stdout, przechwytywane przez $(.funkcja).

funkcje-zwrot.hl 
:pobierz_date def
  > date +%Y-%m-%d
done
data_=$(.pobierz_date)
12

Klasy

Klasy grupują funkcje (metody) w przestrzeń nazw. Definiujemy przez ;;nazwa def i kończymy przez done. Metody klasy są wywoływane jako .nazwa_klasy.metoda. Klasy wspierają dziedziczenie poprzez biblioteki.

klasy.hl 
! Definicja klasy
;;Plik def
  ! Metoda klasy
  :stworz def
    sciezka_="$1"
    > touch $sciezka_
    log "Plik stworzony"
  done
  :usun def
    > rm -f "$1"
  done
done
! Wywołanie metody klasy: .NazwaKlasy.metoda
.Plik.stworz
.Plik.usun

Klasy z polami

Użyj zmiennych lokalnych jako pól klasy.

klasy-pola.hl 
;;User def
  imie_="Anon"
  :ustaw_imie def
    imie_="$1"
  done
done
13

Obsługa błędów: try / catch

Instrukcja try komenda catch komenda_błędu obsługuje błędy w jednej linii. Przydatne do obsługi komend systemowych.

try-catch.hl 
! Prosta obsługa błędu
try curl -s https://api.example.com/data catch echo "Błąd połączenia"
! Operacja na pliku
try cat /etc/config.txt catch echo "Brak pliku konfiguracyjnego"
! Z zapisem do zmiennej
try ./uruchom.sh catch log "Uruchomienie nieudane"

Try w pętlach

Użyj try w pętlach do obsługi błędów w iteracjach.

try-loop.hl 
for url in list_urls > try curl $url catch log "Błąd: $url"
14

Enum (wyliczenia)

Enum definiuje zbiór nazwanych stałych. Składnia: == NazwaEnum [wariant1, wariant2, ...]. Używane w warunkach i structach.

enum.hl 
! Definicja enum
== Status [uruchomiony, zatrzymany, blad, oczekiwanie]
== Kolor [czerwony, zielony, niebieski, zolty]
== Tryb [produkcja, staging, test]

Użycie enum

Enumy mogą być używane w warunkach.

enum-uzycie.hl 
? $status = Status.uruchomiony > log "System aktywny"
15

Struct (struktury danych)

Struct definiuje strukturę z nazwanymi polami i typami. Składnia: struct NazwaStruct [pole:typ, pole2:typ2]. Typy: str, int, bool, enum itp.

struct.hl 
! Definicja struktury użytkownika
struct Uzytkownik [imie:str, wiek:int, aktywny:bool]
struct Serwer [host:str, port:int, protokol:str]
struct Plik [sciezka:str, rozmiar:int, modyfikacja:str]

Instancje struct

Twórz instancje w klasach lub funkcjach.

struct-inst.hl 
user_=Uzytkownik[imie="Jan", wiek=30, aktywny=true]
16

Biblioteki

Biblioteki importujemy przez #<typ/nazwa:wersja>. Dostępne typy bibliotek: source, core, bytes, github, virus, vira. Biblioteki mogą być wersjonowane.

source
Lokalne pliki .hl z ~/.hackeros/hacker-lang/libs/sources/. Dla custom kodu.
core
Biblioteki rdzenia z ~/.hackeros/hacker-lang/libs/core/. Podstawowe funkcje.
bytes
Skompilowane biblioteki .so z libs/bytes/. Dla skryptow i programów (glowne repozytorium pobieraj liby za pomoca bytes install).
github
Biblioteki pobierane z GitHuba (zdalne). Automatyczne pobieranie.
virus
Specjalne biblioteki .a z libs/.vira/. Dla zaawansowanych narzędzi.
vira
Biblioteki pobierane do ~/home/nazwa uzytkownika/.hackeros/hacker-lang/libs/.vira/. Dla języka programownia Hacker Lang od każdego użytkownika.
biblioteki.hl 
! Lokalna biblioteka source
#<source/utils>
#<source/logger>
! Biblioteka core z wersją
#<core/io:2.1>
#<core/http:1.0.3>
! Skompilowana biblioteka
#<bytes/openssl>
! Biblioteka z GitHuba
#<github/user/moja-lib:0.5>

Zarządzanie wersjami

PLSA sprawdza wersje bibliotek podczas analizy.

17

Zależności systemowe

Deklaracja zależności systemowych: // narzędzie. Informuje analizator statyczny (PLSA) jakie narzędzia systemowe są wymagane. PLSA sprawdzi ich obecność.

sysdep.hl 
! Wymagane narzędzia systemowe
// curl
// jq
// git
// ffmpeg
! Dalszy kod może korzystać z tych narzędzi
> curl -s https://api.example.com | jq '.data'

Sprawdzanie zależności

Użyj hl-plsa --check-deps do weryfikacji.

18

Pluginy

Pluginy to zewnętrzne pliki (.hl lub binarne) z katalogu ~/.hackeros/hacker-lang/plugins/. Uruchamiamy przez \\nazwa [argumenty]. Mogą być łączone z sudo. Dany plugin możesz pobrać za pomocą bytes install plugin <nazwa pluginu>

plugins.hl 
! Uruchom plugin "backup"
\\backup
! Plugin z argumentami
\\kompresuj /home/user/dokumenty
! Plugin z sudo (^)
^\\instaluj-pakiety curl git vim
! Plugin z wieloma argumentami
\\deploy staging v1.2.3 --force
ℹ
Plugin może być plikiem nazwa.hl lub binarnym plikiem wykonywalnym nazwa w katalogu pluginów. PLSA sprawdza ich obecność przy użyciu flagi --check-plugins. Twórz własne pluginy dla modularności.
19

Linkowanie zewnętrzne: extern

Instrukcja -- ścieżka deklaruje zewnętrzne linkowanie dynamiczne. -- static ścieżka — statyczne. Używane do integracji z C/C++ libs.

extern.hl 
! Dynamiczne linkowanie
-- /usr/lib/libssl.so
! Statyczne linkowanie
-- static /usr/local/lib/libcustom.a
! Biblioteka systemowa
-- /lib/x86_64-linux-gnu/libc.so.6

Integracja z extern

Użyj w bytes libs.

20

Sudo (^)

Prefiks ^ przed dowolną instrukcją oznacza wykonanie z podwyższonymi uprawnieniami. PLSA oznacza te linie jako potencjalnie niebezpieczne.

sudo.hl 
! Komenda z sudo
^> apt-get update
^> systemctl restart nginx
! Plugin z sudo
^\\instaluj curl git
! Wywołanie funkcji z sudo
^.wyczysc_system
⚡
Ostrzeżenie: Komendy z ^ są wykonywane z prawami root. Analizator PLSA raportuje je jako is_potentially_unsafe = true i zbiera listę ostrzeżeń bezpieczeństwa. Używaj ostrożnie.
21

Lock / Unlock (zmienne zablokowane)

Instrukcje lock i unlock pozwalają blokować i odblokowywać zmienne (muteksy/stałe). Zapobiega przypadkowym zmianom.

lock.hl 
! Zablokuj zmienną (ustaw jako stałą)
lock $WERSJA = "1.0.0"
lock $MAX_POLACZEN = 100
! Odblokuj zmienną
unlock $WERSJA

Lock w funkcjach

Użyj lock do ochrony parametrów.

22

Raw block [ ... ]

Blok surowy [ ... ] zawiera wieloliniowy tekst/kod bez parsowania hacker-lang. Przydatny do heredoc, skryptów inline itp. Zawartość jest wykonywana jako bash.

rawblock.hl 
! Raw block — zawartość nie jest parsowana przez hacker-lang
[
#!/bin/bash
# To jest surowy bash
for i in $(seq 1 5); do
    echo "Liczba: $i"
done
]

Raw do szablonów

Użyj raw do tworzenia plików konfiguracyjnych.

23

Tło (&)

Instrukcja & komenda uruchamia komendę w tle (asynchronicznie). Przydatne do paralelnego wykonania.

background.hl 
! Uruchom serwer w tle
& python3 -m http.server 8080
! Pobieranie w tle
& wget -q https://example.com/plik.zip
! Główny kod kontynuuje działanie
log "Serwer uruchomiony w tle"

Zarządzanie procesami tła

Użyj wait do synchronizacji.

24

Import

Instrukcja << "ścieżka" importuje zasób (plik konfiguracyjny, dane itp.). Wczytuje do środowiska.

import.hl 
! Import pliku konfiguracyjnego
<< "config/app.conf"
<< "/etc/myapp/settings.hl"

Import z URL

Użyj curl do zdalnych importów.

25

Zakończenie programu: end

Instrukcja end [kod] kończy program z opcjonalnym kodem wyjścia. Domyślnie 0.

end.hl 
! Zakończ z kodem 0 (sukces)
end 0
! Zakończ z błędem
end 1
! Zakończ bez kodu (domyślnie 0)
end

End w funkcjach

Użyj end w funkcjach do wczesnego wyjścia.

EX

Przykład: Hello World

Podstawowy przykład demonstrujący zmienne i funkcje.

hello.hl 
! hello.hl — pierwszy program w hacker-lang
@NAZWA="Świat"
:powitaj def
  > echo "Witaj, $NAZWA!"
  log "Program działa poprawnie."
done
.powitaj
end 0
EX

Przykład: Skrypt backupu

Praktyczny skrypt z funkcjami i obsługą błędów.

backup.hl 
! backup.hl — skrypt tworzenia kopii zapasowych
// tar
// gzip
@SRC_DIR="/home/user/dokumenty"
@BACKUP_DIR="/mnt/backup"
:sprawdz_katalog def
  ? ! -d $BACKUP_DIR > mkdir -p $BACKUP_DIR
  log "Katalog backupu gotowy"
done
:zrob_backup def
  data_=$(date +%Y%m%d_%H%M%S)
  plik_="$BACKUP_DIR/backup_$data_.tar.gz"
  try tar -czf $plik_ $SRC_DIR catch log "Backup nieudany!"
  > echo "Backup zapisany: $plik_"
done
:pokaz_rozmiar def
  > du -sh $BACKUP_DIR
done
! ─── Ciało główne ────────────────────
log "=== Backup Start ==="
.sprawdz_katalog
.zrob_backup
.pokaz_rozmiar
log "=== Backup Done ==="
end 0
EX

Przykład: Zaawansowany parser

Skrypt parsujący JSON z API używając jq i struktury.

parser.hl 
// curl
// jq
#<core/json:1.0>
struct Dane [id:int, nazwa:str, wartosc:float]
:pobierz_dane def
  json_=$(> curl -s https://api.example.com/data)
  > echo $json_ | jq '.[]'
done
.pobierz_dane
end 0
