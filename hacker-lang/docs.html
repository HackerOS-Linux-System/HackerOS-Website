HackerOS — język skryptowy

Hacker Lang

Interpretowany język programowania dla HackerOS (Debian-based).
Własna składnia operatorowa, wbudowany linter, REPL, kompilator Cranelift i C runtime.

wbudowany w HackerOS tylko dla HackerOS pliki .hl v0.4 · gen 1
01

Czym jest Hacker Lang?

Hacker Lang (HL) to interpretowany język programowania napisany w Rust, zaprojektowany jako natywny język skryptowy systemu HackerOS. Pliki źródłowe mają rozszerzenie .hl.

HL łączy własną składnię operatorową z bezpośrednim dostępem do powłoki systemowej, wbudowanym linterem, bibliotekami standardowymi (std/net, std/sys…), ponad 40 quick-funkcjami oraz prawdziwym kompilatorem opartym na Cranelift.

✨
Nowe w v0.4: kompilator Cranelift (ELF x86_64 + C runtime), system Genów (using <gen 1>), obsługa shebangu (#!/usr/bin/env hl), operator export (=>), hl exec i hl search.

done

Shebang — #!

Pliki .hl mogą zaczynać się od linii shebang, co umożliwia ich bezpośrednie uruchamianie jako skrypty. Shebang jest opcjonalny.

💡
Ważne: #! to shebang — rozpoznawany tylko w pierwszej linii. Operator # w pozostałych miejscach to import biblioteki (# <std/net>). Nie ma konfliktu.
SKŁADNIA SHEBANG
#!/usr/bin/env hl         # zalecana forma
#!/usr/bin/hl             # bezpośrednia ścieżka
#!/usr/bin/env hl --verbose # z argumentami
BASH — uruchamianie jako skrypt
chmod +x skrypt.hl
./skrypt.hl              # uruchom bezpośrednio
05

System Genów — using

Geny to odpowiednik Rust editions — wydawane co 3 lata, definiują dostępną składnię i semantykę języka. Deklaracja jest opcjonalna — bez niej przyjmowany jest gen 1 (domyślny).

🧬
Aktualny maksymalny gen: gen 1. Nowe geny będą dodawane co 3 lata wraz z rozwojem języka. Każdy gen może dodać nowe operatory lub zmienić semantykę — z zachowaniem kompatybilności wstecznej w ramach gena.
DEKLARACJA GENA
#!/usr/bin/env hl         # shebang (opcjonalny)
using <gen 1>             # gen (opcjonalny, domyślnie: gen 1)
/// Opis skryptu

;; ... kod ...
GenStatusOpis
gen 1aktywnyPełna składnia podstawowa, export, shebang, kompilator Cranelift
gen 2zarezerwowanyAsync komendy, typowane zmienne (planowane)
gen 3zarezerwowanyDomknięcia (planowane)
INFO O GENIE PLIKU
hl gen-info skrypt.hl     # wyświetl gen, shebang i liczbę węzłów AST
hl check --meta skrypt.hl # sprawdź z meta-informacjami

Wyjście — ~>

Operator ~> to jedyny sposób wypisywania tekstu na stdout w Hacker Lang. Używanie echo wewnątrz bloków komend jest zakazane i generuje błąd.

🚫
echo jest zabronione. Użycie > echo tekst generuje błąd. Linter automatycznie sugeruje zamianę na ~> tekst. Jeśli potrzebujesz echo w skrypcie powłoki, użyj > bash -c "echo ...".
SKŁADNIA ~>
~> Zwykły tekst
~> Wartość zmiennej: @nazwa
~> Host: @target, Port: @port
~> Wersja HL: @HL_VERSION
~>              ;; pusta linia (lub ::nl)

08

Komendy — > ^ ->

Hacker Lang oferuje siedem trybów uruchamiania komend systemowych:

> komenda

Zwykłe wykonanie. Dziedziczy środowisko.

^> komenda

Wykonanie z sudo. Natywny odpowiednik.

-> komenda

Izolowany namespace Linux (unshare). Oddzielny mount/pid/net.

^-> komenda

Izolacja namespace + sudo.

>> komenda @var

Komenda z interpolacją zmiennych @nazwa.

^>> komenda @var

Interpolacja + sudo.

->> komenda @var

Interpolacja + izolacja namespace.

PRZYKŁADY KOMEND
% target = 192.168.1.1
% port   = 22

>   ls -la /tmp
^>  apt-get update
->  curl -s http://test.internal
^-> nmap -sS 10.0.0.0/24
>>  nmap -p @port @target
^>> ssh @user@@target
->> curl -s http://@target

;; Przekierowania i potoki — normalna składnia powłoki:
> ls -la | grep ".hl"
> cat /etc/passwd | cut -d: -f1 > /tmp/users.txt
> bash -c "printf 'tekst' > /tmp/plik.tmp"
🔒
Izolacja namespace: tryby -> i ^-> używają unshare --mount --pid --net --fork. Komenda uruchamiana jest w oddzielnych przestrzeniach nazw Linuksa, bez dostępu do sieci hosta.
💡
Linter: użycie > sudo cmd jest automatycznie wykrywane i sugeruje zamianę na natywny operator ^> cmd.

Operator % deklaruje zmienną lokalną HL — widoczną tylko w środowisku interpretera. Operator @ odwołuje się do wartości zmiennej.

TYPY WARTOŚCI

% port    = 8080               ;; liczba
% debug   = true               ;; bool
% path    = /tmp/audit/@host   ;; interpolowany string

~> Skanowanie @host:@port → @path

Export — =>

Operator => ustawia zmienną w środowisku procesu (setenv). Wszystkie uruchamiane podprocesy dziedziczą tę zmienną. Różni się od % — zmienna lokalna HL nie jest widoczna dla podprocesów.

POJEDYNCZA WARTOŚĆ
=> EDITOR   = nvim
=> BROWSER  = firefox
=> JAVA_HOME = /usr/lib/jvm/java-17
=> GOPATH   = /home/hacker/go
LISTA WARTOŚCI (PATH)
=> PATH [
| /usr/local/bin
| /usr/bin
| /usr/lib/HackerOS
| /home/hacker/.cargo/bin
| /home/hacker/.local/bin
]
💡
Lista wartości jest łączona znakiem : (konwencja PATH i podobnych). Elementy listy obsługują interpolację @zmiennych.
LISTA Z INTERPOLACJĄ
% home = /home/hacker

=> PATH [
| @home/.local/bin
| @home/go/bin
| @home/.cargo/bin
| /usr/local/bin
| /usr/bin
]

=> EDITOR = nvim
=> GOPATH = @home/go

;; Teraz podprocesy widzą PATH, EDITOR, GOPATH
> bash -c "echo $PATH"
OperatorZasięgDziedziczenie
% name = valTylko HL (lokalna)Podprocesy NIE widzą
=> NAME = valŚrodowisko procesuPodprocesy WIDZĄ (setenv)

Funkcje — : i --

Funkcje definiuje się operatorem :, po którym następuje nazwa i słowo def. Ciało funkcji kończy się słowem done. Wywołanie to -- nazwa.

SKŁADNIA FUNKCJI
: perform_audit def
    ~> Rozpoczynam skan @target...
    ::hr 40

    >> nmap -sV -T4 @target

    ? ok
        ::green Skan zakończony pomyślnie.
    done

    ? err
        ::red Błąd podczas skanowania!
    done
done

;; Wywołanie:
-- perform_audit
12

Logika — ? ok / ? err

Bloki warunkowe sprawdzają last_exit — kod wyjścia ostatnio wykonanej komendy. ? ok uruchamia blok gdy exit = 0, ? err gdy exit ≠ 0.

PRZYKŁADY LOGIKI
> ping -c 1 192.168.1.1

? ok
    ::green Host online ✓
    >> nmap -sV @target
done

? err
    ::red Host niedostępny ✗
    ~> Sprawdź połączenie sieciowe.
done

;; Quick-funkcje też ustawiają exit code:
::contains @text error

? ok
    ~> Znaleziono błąd w tekście
done

;; Zagnieżdżanie:
::exists /etc/passwd
? ok
    ::exists /etc/shadow
    ? ok
        ::green Oba pliki istnieją
    done
done

hl exec i hl search

Skrypty systemowe HackerOS są przechowywane w /usr/share/HackerOS/Scripts/Bin/. Komendy hl exec i hl search ułatwiają pracę z nimi.

hl search
hl search all                # wylistuj wszystkie skrypty
hl search update             # skrypty zawierające "update"
hl search kernel             # skrypty dotyczące kernela
hl exec
hl exec update-system        # uruchom update-system.hl
hl exec check-updates-notify # sprawdź aktualizacje
hl exec unpack-xanmod        # zainstaluj kernel XanMod
💡
hl search automatycznie czyta opis skryptu z pierwszego komentarza /// lub ;; — dlatego warto dokumentować skrypty.

Kompilator — Cranelift + C Runtime

HL posiada prawdziwy kompilator oparty na Cranelift (nie LLVM, nie rustc). Kompilator generuje statyczną binkarkę ELF x86_64 lub bibliotekę .so (ekosystem Virus). Runtime C jest wbudowany w binkarkę kompilatora — użytkownik nigdy go nie widzi.

KOMPILACJA
hl compile skrypt.hl           # → statyczna binarka ELF
hl compile skrypt.hl -o mojprog # → niestandardowa nazwa
hl compile --shared wirus.hl   # → wirus.so (ekosystem Virus)
hl compile --opt none skrypt.hl # → bez optymalizacji
hl compile --opt size skrypt.hl # → optymalizacja rozmiaru
hl compile -v skrypt.hl        # → verbose (pipeline steps)
⚙
Pipeline kompilatora: .hl → AST → HlIR → Cranelift IR → .o + hl_runtime.o → ELF/SO

Diagnostyka i linter

HL posiada wbudowany linter wyświetlający błędy w stylu Rust — z numerami linii, podkreśleniami i sugestiami naprawy.

  --> update.hl:8:1

Sprawdzanie wersji i aktualizacja

check-version.hl
/// Porownaj wersje lokalna z zdalna

// curl

% LOCAL_JSON  = /usr/share/myapp/version.json
% REMOTE_URL  = https://example.com/version.txt
% LOCAL_TMP   = /tmp/local_ver.tmp
% REMOTE_TMP  = /tmp/remote_ver.tmp
% CHECK_TMP   = /tmp/check_ver.tmp

;; Odczyt wersji lokalnej
>> grep version @LOCAL_JSON | grep -o "[0-9][0-9.]*" | head -1 > @LOCAL_TMP

;; Pobierz zdalna
>> curl -fsSL -o @REMOTE_TMP @REMOTE_URL

;; Porownaj sort -V (semantyczne wersjonowanie)
>> cat @LOCAL_TMP @REMOTE_TMP | sort -V | tail -n1 > @CHECK_TMP

;; Czy zdalna jest wieksza?
>> diff -q @REMOTE_TMP @CHECK_TMP > /dev/null 2>&1

? ok
    >> diff -q @LOCAL_TMP @REMOTE_TMP > /dev/null 2>&1
    ? ok
        ::green Aplikacja jest aktualna!
    done
    ? err
        ::yellow Dostepna nowa wersja!
    done
done

>> rm -f @LOCAL_TMP @REMOTE_TMP @CHECK_TMP

Przykłady — sieć i audyt

Pełny audyt sieciowy

network_audit.hl
/// Network Audit Tool — HackerOS

// nmap
// curl
// whois
# <std/net> | <ports>
# <std/sys>

% target     = 192.168.1.1
% output_dir = /tmp/audit
% ports      = 1-1024

::hr 50
::bold Network Audit — HackerOS
~> System: @SYS_OS | Kernel: @SYS_KERNEL
::hr 50

: check_host def
    ~> Sprawdzam: @target
    >> ping -c 1 -W 2 @target
    ? ok
        ::green @target ONLINE ✓
    done
    ? err
        ::red @target nie odpowiada ✗
    done
done

: port_scan def
    ~> Skanuję porty @ports na @target...
    >> nmap -sV -T4 -p @ports @target
    ? ok
        ::green Skan zakończony.
    done
done

: isolated_probe def
    ~> Sondowanie w izolacji sieciowej...
    ->> curl -s --max-time 5 http://@target
done

^> mkdir -p @output_dir

-- check_host
-- port_scan
-- isolated_probe

::nl
~> Audyt zakończony. Wyniki: @output_dir
